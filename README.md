# alerting-system

**Репозиторий:** [github.com/EDINDANDIL/alerting-system](https://github.com/EDINDANDIL/alerting-system)

Платформа для **автоматического скрининга рыночных событий** на большом потоке данных: пользователь задаёт гибкие правила (тип сценария, биржа, рынок, чёрный список инструментов и т.д.), система в реальном времени отслеживает поток и при срабатывании условия выдаёт **сигнал с низкой задержкой** (целевой ориентир для контура — до **~100 мс** end-to-end в рабочем профиле нагрузки). Дальше сигнал может уходить в **торговые терминалы** (Tiger Trade, MetaScalp, Vataga), во **внешние алгосервисы** или напрямую пользователю — в зависимости от интеграции.

Сейчас в репозитории — **MVP на микросервисной архитектуре**: ядро обработки и контракты заложены так, чтобы расширять число типов фильтров и каналов доставки без ломки модели.

---

## Зачем это нужно

**Проблема.** Ручной мониторинг биржи на потоковых данных плохо масштабируется: объём событий огромный, а часть сценариев требует реакции за **секунды** или меньше. Без автоматизации трейдер тратит внимание на «просмотр ленты» вместо исполнения стратегии и рискует **пропускать входы** или реагировать с запозданием.

**Задача.** Дать пользователю возможность один раз настроить правила и получать **своевременный сигнал** в момент появления интересующей ситуации на рынке: снижается когнитивная нагрузка, повышается скорость и качество решений при сохранении контроля рисков. В типичном сценарии: **произошло событие по правилам пользователя → сработал триггер →** открывается окно/виджет в терминале по инструменту **или** внешний сервис выполняет заранее заданное действие.

**Отличие от типовых аналогов** (Telegram-боты, агрегаторы вроде [scalpboard.io](https://scalpboard.io/app)): акцент — на **гибкости конфигурации фильтра** и на **интеграции в торговый контур** (терминалы и алгоритмы), а не только на уведомлении в мессенджере.

## Как устроено решение (продуктовый взгляд)

- **Оркестрация фильтров** — приём и хранение пользовательских конфигураций, учёт подписок, надёжная доставка изменений в поток обработки.
- **Потоковая обработка** — отдельный сервис потребляет рыночные данные и события фильтров, держит состояние правил и вычисляет срабатывания (в MVP — сценарий **IMPULSE** на скользящем окне).
- **Доставка сигналов** — отдельный контур для маршрутизации алертов в терминалы, алгосервисы или другие каналы (в текущем коде — этап **в разработке**, см. ниже).

Интеграция с рынком: обработанные рыночные данные поступают из **внешнего парсера на C++**, который по согласованному контракту публикует сообщения для Java-контура (например, в Kafka).

## Что уже есть в коде и что в планах

| В целевой архитектуре                                  | В этом репозитории сейчас |
|--------------------------------------------------------|---------------------------|
| Модуль оркестрации фильтров                            | `filter-service` — HTTP API, PostgreSQL, outbox |
| Надёжная доставка событий фильтров                     | `outbox-sender` + паттерн **Transactional Outbox** |
| Потоковый процессор                                    | `data-processor` — Kafka consumers, состояние фильтров, логика IMPULSE |

| Авторизация и полноценный мультипользовательский режим | **Планируется** (сейчас идентификация пользователя — заголовок `X-user-id`) |

## Модули (детальнее)

### `filter-service`

Принимает запросы с конфигурацией фильтра (например, **IMPULSE**: биржи, рынки, чёрный список, окно времени, направление, порог и т.д.), сохраняет данные в свои таблицы. Параллельно каждое значимое изменение (создание/удаление, подписка/отписка) попадает в **outbox** в той же транзакционной логике, чтобы downstream не терял события при сбоях.

События должны обрабатываться в **согласованном порядке** (например, корректная последовательность операций CREATE → SUBSCRIBE → … → DELETE), а в Kafka сообщения **партиционируются по ключу**, чтобы сохранить порядок обработки на стороне consumer и упростить идемпотентность.

### `outbox-sender`

Планировщик, который читает outbox и отправляет события в Kafka (`filter-topic`) в транзакции с БД: успешная отправка — запись из outbox удаляется; ошибка — откат, повторная попытка позже.

### `data-processor`

Потоковый процессор: отдельные Kafka-топики под **рыночные данные** (`trades-topic`) и под **события фильтров** (`filter-topic`), разные настройки consumer groups. Поддерживается состояние фильтров и логика обнаружения сигналов (IMPULSE на скользящем окне). **Целевой** шаг — публикация алерта в Kafka для последующего `alert-service`; в текущей версии MVP см. ограничения ниже.

### `alert-service` (план)

Оркестрация доставки сигналов: маршрутизация в торговый терминал, во внешний алгоритм или другой канал. В репозитории пока не выделен отдельным модулем.

### Общее

Через API задаётся фильтр и параметры; в реальном времени идёт проверка по потоку данных; при триггере формируется сигнал для downstream. Количество типов фильтров в MVP ограничено, но **модель и границы сервисов** рассчитаны на расширение до продуктового уровня.

## Структура репозитория

- `filter-service` - HTTP API + запись фильтров и outbox в PostgreSQL
- `outbox-sender` - фоновый sender из outbox в Kafka
- `filter-outbox-common` - общие DTO/мапперы/outbox-модель
- `data-processor` - отдельный Gradle-проект для обработки Kafka-событий
- `KafkaClusterDocker-compose.yaml` - локальный Kafka-кластер (3 брокера)

## Поток данных (e2e)

1. Клиент вызывает `POST /api/filters` (или `DELETE /api/filters`) в `filter-service`.
2. `filter-service` обновляет таблицы фильтров и пишет событие в `filter_outbox`.
3. `outbox-sender` (каждые 50ms) читает `filter_outbox` в транзакции и публикует в Kafka `filter-topic`.
4. После успешной отправки запись outbox удаляется.
5. `data-processor` читает:
   - `filter-topic` -> обновляет внутреннее состояние фильтров;
   - `trades-topic` -> прогоняет trade через процессор.
6. Алерты сейчас отправляются в лог (`AlertPublisher`), без отдельного Kafka producer.

## Быстрый старт

### 1) Поднять Kafka

```bash
docker compose -f KafkaClusterDocker-compose.yaml up -d
```

### 2) Подготовить PostgreSQL

Сервисам нужны таблицы:

- `impulse_filters`
- `user_impulse_filters`
- `filter_outbox`

В репозитории нет полноценных SQL-миграций для всех таблиц, поэтому схему нужно создать вручную или через ваш migration tool.

### 3) Собрать проект

Из корня:

```bash
gradle build
```

Для `data-processor` можно использовать wrapper:

```bash
cd data-processor
./gradlew build
```

### 4) Запустить сервисы

В отдельных терминалах:

```bash
gradle :filter-service:run
```

```bash
gradle :outbox-sender:run
```

```bash
cd data-processor
./gradlew run
```

## API (`filter-service`)

### POST `/api/filters`

Создает/привязывает фильтр к пользователю. Требуется заголовок `X-user-id`.

Пример:

```http
POST /api/filters
X-user-id: 101
Content-Type: application/json

{
  "action": "IMPULSE",
  "exchange": ["binance"],
  "market": ["futures"],
  "blackList": ["btcusdc"],
  "timeWindow": 60,
  "direction": "UP",
  "percent": 2,
  "volume24h": 1000000
}
```

Ответ: `200 ok`

### DELETE `/api/filters`

Отписывает пользователя от фильтра. Тело и заголовки аналогичны `POST`.

Ответ: `200 ok`

## Текущие ограничения MVP

- Полный Docker Compose для всех Java-сервисов и PostgreSQL пока не добавлен.
- Алерты в `data-processor` пока логируются, а не публикуются в отдельный topic.
- SQL-миграции для всех таблиц в репозитории пока неполные.

## Диагностика проблем

- Нет событий в `filter-topic`:
  - проверить запуск `outbox-sender`;
  - проверить наличие записей в `filter_outbox`;
  - проверить доступность Kafka bootstrap-серверов.
- `POST /api/filters` падает:
  - проверить заголовок `X-user-id`;
  - проверить наличие и корректность таблиц в PostgreSQL.
- `data-processor` не реагирует на трейды:
  - проверить, что сообщения действительно приходят в `trades-topic`;
  - проверить `group.id` и `auto.offset.reset` в `data-processor/application.conf`.
